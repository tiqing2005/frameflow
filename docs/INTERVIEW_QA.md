# FrameFlow AI 面试问答手册

## 使用方式

不要背诵整段答案。每个问题应按“结论→证据→取舍→扩展方向”回答，并能在项目页面、运行记录、测试或代码中指出对应证据。如果最终实现和本文不一致，以实际运行结果为准并在提交前修正本文。

## 1. 选题与范围

### Q1：为什么选择题目二，不选 Android Agent？

**答：**题目二能在 2～3 天内把评分重心集中在业务闭环、持久化、故障恢复和方案解释上。Android 题除了 Web 后台，还有特定 API 31 时钟 App、无障碍树差异、新模拟器验收和 APK 交付风险。这不代表题目二简单，而是其主要风险更可在公网环境中自动验证。

**证据：** `SCORING_MATRIX.md` 将每个必须项映射到 API/UI/测试。

### Q2：为什么不做完整剪辑器或专业视频导出？

**答：**原题重点是“输入、字幕处理、素材匹配、人工调整、结果保存”。本版已经用持久化异步任务和 ffmpeg 生成可播放的 MP4 组合预览，时间线按字幕时间戳或安全估算时长组织素材；但它仍不是帧级非线性编辑器，也不提供多音轨、复杂转场、码率/分辨率模板等专业导出参数。这样既能验证真实媒体闭环，又不把 Demo 夸大成完整剪辑软件。

## 2. 技术选型

### Q3：为什么选 FastAPI + React，而不是 Next.js 全栈？

**答：** React/Vite 足以快速完成编辑器交互，Python 对 ffmpeg、中文 NLP、TF-IDF、ASR/Embedding 适配更直接。更关键的是纯 Next.js Serverless 并不适合本地 SQLite、持久化上传和长任务。如果改成常驻 Next Server + Node Worker 当然可以，但不再有 Serverless 的简单性，还降低了 AI/媒体开发效率。

### Q4：为什么选择模块化单体，不拆微服务？

**答：**当前领域小、交付时间短，Project/Job/Segment/Recommendation 有很强的事务一致性。拆服务会立即引入队列、分布式追踪、对象存储和数据最终一致性，但没有对应吞吐量需求。我保留了清晰模块边界与 Provider 接口，需要规模化时再拆 Worker/Provider。

### Q5：为什么使用 SQLite？它是“生产级”吗？

**答：**它对单实例 Demo 是合适的持久化数据库：事务、外键、唯一约束、WAL 和备份方案都是真实的。但我不宣称它适合多机水平扩容或高写并发。当前通过单 Worker、WAL、busy timeout 和短事务控制范围；规模化时迁移 PostgreSQL。

### Q6：为什么不上 Redis/Celery？

**答：**任务与项目的创建需要原子性。如果业务数据在 SQLite、队列在 Redis，会有“已建项目但未入队”的双写问题，还需要 outbox/reconciler。当前把 Job 放在同一数据库可在一个事务完成。吞吐量增长时，正确扩展不是直接双写 Redis，而是 PostgreSQL outbox + 队列。

## 3. 异步任务与稳定性

### Q7：为什么不用 FastAPI `BackgroundTasks`？

**答：** `BackgroundTasks` 和 API 进程生命周期绑定，进程重启、容器调度或崩溃时无法恢复，也很难保存租约、进度事件和人工重试。本项目先持久化 Job，独立 Worker 原子领取，所以 API 请求结束不影响处理。

### Q8：Worker 在任务中途崩溃怎么办？

**答：** Job 领取时写入 `lease_owner/lease_expires_at/heartbeat_at`。Worker 执行时续租；崩溃后租约过期，启动恢复或下一轮领取将其重新排队，达到最大次数则失败。为了安全重执行，分段/推荐不是无条件 append，而是事务化换版/upsert。

### Q9：两个 Worker 会不会抢到同一任务？

**答：**当前部署边界是单 Worker，先从运维上避免无需要的竞争。数据模型仍保留原子领取和租约所有者，但我不因此声称已验证多 Worker 高并发。扩展时应在 PostgreSQL 使用 `FOR UPDATE SKIP LOCKED` 或专用队列，并增加并发与崩溃测试。

### Q10：任务进度是真的吗？

**答：**进度是对处理阶段的离散估计，不是对未知外部服务剩余时间的精确预测。但它不是前端假计时：Worker 每进入一个真实阶段都会持久化 stage/progress/JobEvent，前端轮询后显示。

### Q11：怎样处理重复创建？

**答：**客户端为一次创建生成 `Idempotency-Key`。后端用 `(scope,key)` 唯一约束，同时保存规范化请求 hash。相同 Key + 相同内容返回原项目/任务；相同 Key + 不同内容返回 409，防止将两个业务意图错误合并。

### Q12：取消一个 running 任务能够立即停止吗？

**答：**不能无条件保证立即中断底层系统/第三方调用。取消是合作式的：设置状态/取消意图，Worker 在阶段边界和安全检查点停止。对外部 HTTP 调用另外需要超时，不能只靠取消标志。

## 4. 字幕与匹配

### Q13：字幕分段算法是什么？

**答：**默认主干是确定性规则：按中英文句末符号和换行切分，合并过短句，按最大长度再切过长段。关键词通过中文分词/频率与停用词提取。它不会做深层语义重写，但对新文本可重现、可测，且无外部服务时不中断业务闭环。

### Q14：你的匹配算法是 AI 吗？

**答：**默认实现是本地信息检索/机器学习方法，不是外部大模型。字符 n-gram TF-IDF 会把文本变成稀疏向量并计算余弦相似度，所以它的确是向量相似度；但它不是神经语义 Embedding。运行记录会如实标记 `rules/hybrid-tfidf-v1`。

### Q15：为什么用字符 n-gram，不只用 jieba 词？

**答：**中文无显式词边界，新领域词、产品名和组合词可能被分词器错分。字符 2～4 gram 可降低这个依赖，对部分重合更鲁棒。但它会高估表面相似，所以我没有只用这一项，而是再加关键词和标签/主题分项。

### Q16：0.55/0.30/0.15 是怎么定的？

**答：**这是 Demo 小样本上的可解释初始权重，不宣称是经过大规模学习的最优值。TF-IDF 承担主要文本相似度，关键词给明确概念更强证据，标签/主题用于领域信息。正式优化时应构建“片段–素材相关级别”标注集，用 NDCG/Recall@3 和人工选择数据调整权重。

### Q17：为什么必须返回 3 个？没有足够相关素材怎么办？

**答：**这是原题的显式验收要求，也符合“候选辅助人工决策”的产品逻辑。系统先按相关性排序唯一素材；高相关不足时用多样性/稳定顺序补齐，但保存 `is_diversity_filler=true` 并在 UI 显示“低相关补位”，不伪装高相关。

### Q18：用户修改字幕后候选会怎么办？

**答：**修改文本/主题/关键词先用版本号持久化。候选与输入版本有关，所以需要显示重新匹配入口或标记候选已过期。局部 rematch 会事务化换候选，但保留 `manual` Selection。这样系统可更新建议，却不替用户重做决定。

## 5. 持久化、一致性与 API

### Q19：为什么 Segment 要有 `version`？

**答：**两个页面可能同时打开同一项目。如果不带版本，后保存的旧内容会静默覆盖先保存的新内容。PATCH 必须携带客户端所见 `version`，不匹配返回 409 和当前版本，UI 提示重新加载，不自动覆盖。

### Q20：拖拽排序怎样避免中间唯一冲突？

**答：**接口传入完整 `segment_ids` 顺序，后端首先验证不重复、不缺失且都属于该项目。因为有 `(project_id,position)` 唯一约束，实现在同一事务先将位置移到不会冲突的临时偏移区，再按新顺序写回 0..N-1。

### Q21：为什么 JSON 数组存 Text，不做关系标签表？

**答：**当前标签/关键词数量小，主要作为推荐解释和小库搜索元数据。规范 JSON Text 可保持 SQLite 便携性并降低表数，业务边界会去重、限长并转成 API 数组。如果需要大规模标签统计、同义词、权限或复杂组合筛选，应迁移为关系标签表/JSONB 索引。

### Q22：项目删除会不会删掉素材？

**答：**不会。Project 是内容处理聚合，Asset 是全局素材库。删除 Project 级联删除其 Source/Jobs/Segments/Recommendations/Selections，但不删除 Asset。Selection 对 Asset 是 RESTRICT，素材通常用 active 软禁用，保留历史引用。

## 6. AI、ASR 与真实边界

### Q23：你到底有没有接真实 LLM？

**答：**默认交付版没有把外部 LLM 作为运行时必须依赖。实际分段和匹配是规则 + TF-IDF 混合检索，运行记录显示 `provider=rules`。不会把开发期间使用 Codex 与产品运行时调用 LLM 混为一谈。

### Q24：那为什么产品名里有 AI？

**答：**产品解决的是“智能分析/推荐”场景，而 AI 要求允许规则、Embedding、LLM 或混合方案。不过品牌名不能代替技术披露，所以 UI 运行记录和 `AI_USAGE.md` 都明确写默认为 rules/TF-IDF。

### Q25：音频/视频转写是否已完成？

**答：**默认环境中，真实上传、存储和异步任务已存在，但无 ASR Key 时任务会如实以 `ASR_NOT_CONFIGURED` 失败，不伪造字幕。题目明确允许保留文本稳定入口，同时说明真实接入方案。Provider 边界是 `Transcriber.transcribe(file)`，正式接入还需 ffmpeg 音轨提取、超时、文件时长/费用限制和真实音频测试。

### Q26：`ai_degrade` 真的在模拟外部 AI 故障吗？

**答：**它是明确标记的一次性故障注入，用来可重复验收“降级标记、运行追溯和成功状态”路径，不证明某个真实 Provider 已崩溃。`job_fail` 同理，是为了可重复验收失败/重试，不伪造外部调用历史。

### Q27：真实 AI 服务失败你会怎么处理？

**答：**先分类：429/5xx/连接超时是暂时性，可有限次指数退避；401/403/配置缺失不自动重试；结构化输出不合法先做 Schema 校验和有限修复。字幕语义增强/匹配失败可回退规则引擎并记录 degraded；ASR 失败不能用规则“猜”语音内容，应失败并重试/让用户粘贴文本。

## 7. 安全、部署与规模化

### Q28：怎样保证 API Key 不泄露？

**答：** Provider Key 只从后端环境变量/密钥管理服务读取，不以 `VITE_` 变量出现，不经 API 返回，不写 SQLite，不写 Audit/JobEvent。`.env` 被 gitignore，只提交 `.env.example` 占位符。发布前还应搜索前端构建产物和 git 历史，不只检查当前文件。

### Q29：上传文件有哪些风险？

**答：**路径穿越、MIME 伪造、大文件磁盘/内存耗尽、解码炸弹、公网滥用和残留孤儿文件。当前最小缓解是大小限制、允许类型、服务端随机存储名、持久卷与上传配额。但扩展到不可信公网用户时还需隔离解码容器、真实文件类型检测、恶意文件扫描、限流和对象存储。

### Q30：为什么不能把这个项目部署到 Vercel 就结束？

**答：**因为核心边界是持久化 SQLite、持久化上传文件和常驻 Worker。典型 Serverless 函数的临时文件系统和请求时限不满足。应使用可挂载 `/data` 持久卷的单容器/单机，同时运行 API 和 Worker；或迁移到 PostgreSQL + 对象存储 + 队列后再使用多实例平台。

### Q31：如果素材从 12 个增加到 100 万个，怎么改？

**答：**不再对全库内存计算。先用标签/权限/类型过滤和全文检索取关键词候选，再用向量 ANN 取语义候选，使用 RRF 或学习排序融合并二次重排。Embedding 要按模型/维度/输入 Hash 版本化，素材新增通过异步索引管线，还要有重建索引、召回率测试和权限过滤，不能只是换一个向量库名字。

## 8. 测试、调试与 AI 辅助开发

### Q32：你测了什么？

**答：**分三层：单元测试覆盖分段、关键词、混合排序/补位和幂等等确定性逻辑；API 集成测试覆盖创建→Worker→项目结果→编辑→选择→重读，以及故障/重试、版本冲突；浏览器/验收脚本覆盖真实运行环境的健康、素材、幂等创建和最终状态。具体结果必须以提交 commit 的真实执行记录回答，不只说“有测试文件”。

### Q33：现场出错你会怎么查？

**答：**先从用户可见证据定位层次：

1. `/health/live` 判断 API 进程，`/health/ready` 判断 DB/Worker。
2. 从 UI 复制 request_id/job_id，查 Job 当前 stage 和事件最后一条。
3. queued 长时不动：查 Worker 心跳、数据目录是否一致。
4. running 卡住：查租约/心跳、当前外部调用超时和系统资源。
5. succeeded 但页面空：直接 GET Project 判断是 API 聚合问题还是前端解析问题。
6. 数据异常：保留现场、备份 DB，通过 Audit/JobEvent 还原，不先手工改库破坏证据。

### Q34：AI 帮你开发了哪些部分？

**答：**需求拆解、架构备选、部分代码/UI 初稿、测试用例和交付文档使用了 Codex 辅助。需要主动区分“开发过程用 AI”与“产品运行时用 AI”：当前运行时默认为规则/TF-IDF，不依赖外部 LLM。候选人必须亲自复跑测试、检查公网流程并能解释关键决策，不能将“AI 生成了代码”当作免责理由。

### Q35：如果再给你一周，最优先做什么？

**答：**优先级不是预览特效，而是：

1. 接入并实测真实 ASR，完成 ffmpeg 音轨提取、时长/费用限制、超时与时间戳。
2. 增加一小批人工相关性标注，用 Recall@3/NDCG 评估并调整混合权重。
3. 对公网上传增加访问控制、限流、更强文件检测与数据清理。
4. 引入 Alembic、备份/恢复演练与更完整的浏览器自动化。
5. 在可靠时间戳基础上再做简单时间轴/预览，不把视觉加分项提前到数据正确性之前。

## 9. 最后的 30 秒概括

> 这个项目的主要决策是把“可靠且可解释”放在“外部 AI 看起来很强”之前。API 不执行长任务，而是事务内创建持久化 Job；Worker 用租约恢复；匹配保存分项证据；人工选择永远优先；无 ASR/LLM 时不伪造结果。它的明确边界是单实例 SQLite 和单 Worker，扩展路线是 PostgreSQL、对象存储和专用队列。
