# FrameFlow AI 已知问题与取舍

## 1. 文档目的

本项目的完成时间受限，以核心业务闭环和故障可恢复性为优先级。下列内容是已知边界，不会在演示中隐藏或包装成已完成能力。

## 2. 功能边界

### KI-01：本地 ASR 受单机资源限制

- **影响**：公网使用 `faster-whisper small/int8` 在专用服务器本地推理，转写会占用主要 CPU；长音频和排队任务不适合承诺固定完成时间。首次启动还需要下载模型。
- **当前处理**：生产配置固定单 Worker，容器分配 3.5 CPU / 4 GB，模型缓存持久化到 `/data`。一次 71 秒热机测试音频的 ASR 阶段约 20.5 秒、完整流程约 26 秒，但只作为样本记录而非 SLA。无本地依赖或可用 Provider 时仍会以 `ASR_NOT_CONFIGURED` 失败，不生成假字幕。
- **可选方案**：保留 OpenAI-compatible ASR 与 DashScope Paraformer-v2。DashScope 路径先生成 8 kbps MP3、按 75 秒切片，再用 HMAC 临时 URL 回源；该优化降低传输量，但不能保证新加坡到国内云服务的跨境链路稳定。
- **后续方向**：补充长音频、噪声样本、冷启动与并发队列基准；资源增长后将 ASR 拆到专用推理 Worker。

### KI-02：默认匹配可降级为字符相似，需启用 Embedding 才获得语义泛化

- **影响**：在未安装/配置 Embedding provider 时，0.55 权重通道退化为字符 n-gram TF-IDF，对同义改写、比喻和词表外的新字幕匹配有限。
- **当前处理**：新增 `embeddings.py` 语义 Provider 边界：默认 `EMBEDDING_PROVIDER=auto`，安装 `requirements-embeddings-local.txt`（本地 BGE `bge-small-zh-v1.5`）或配置远程 `/embeddings` 后自动启用真·向量相似度；任何失败回退字符相似，保证零依赖可演示。`evaluation/` 在 10 条 hard case（同义/比喻/干扰项）上测得 Hard Hit@3 从 0.60（字符）提升到 0.80（向量），证明语义通道有效。
- **后续方向**：对更大素材库引入向量索引（pgvector/Milvus），并用更多标注样本调整权重。

### KI-03：自动标签目前主要依据文本元数据，不做视觉内容识别

- **影响**：上传文件只有名称和用户输入可供分析时，无法识别图片中的真实物体、人物或视频关键帧内容。
- **当前处理**：上传素材时若用户未填 tags/keywords，会自动调用 `llm.suggest_asset_tags` 生成中文标签与画面关键词；LLM 未配置或失败时回退规则提取（`extract_keywords`）。每次自动标签均记录 `asset_tagging` AIRun，预置素材使用人工复核的中文元数据。
- **后续方向**：对图片抽帧/视频关键帧调用视觉模型，生成画面标签后由人工确认。

### KI-04：组合预览不是专业非线性编辑器

- **影响**：当前时间线按字幕片段组织匹配素材，不提供帧级裁切、复杂转场、多音轨混音或专业导出参数。
- **当前处理**：工作台已提供按片段时长展示的素材时间线、片段定位、异步 MP4 生成与最终播放。Worker 会标准化图片/视频、拼接片段并在 ffmpeg 支持时烧录中文字幕；渲染请求按时间线指纹幂等，重复、失败与重试状态可追踪。
- **后续方向**：在稳定词级时间戳基础上增加裁切点、转场、音轨与可配置导出模板，同时继续保持预览队列与业务编辑解耦。

## 3. 工程边界

### KI-05：SQLite 和单机有界 Worker 池不适合水平扩容

- **影响**：当前公网固定单 Worker，ASR 运行时后续任务会排队；吞吐量仍受 SQLite 单写者、容器资源和单持久卷限制，不应同时启动多个跨机写实例。
- **当前处理**：WAL、`busy_timeout`、持久化队列、原子领取、独立 Worker 心跳、租约恢复与有界进程池，完成单机低并发下的可靠背压。单 Worker 是本地 ASR 避免多模型进程争抢资源的稳定性取舍，不代表代码只能运行一个 Worker。
- **后续方向**：PostgreSQL + 业务 outbox + Redis/RabbitMQ 队列 + 对象存储，再进行横向 Worker 扩容。

### KI-06：上传文件使用本地持久卷

- **影响**：只能部署为可挂载持久磁盘的单实例；磁盘清理会造成素材与数据库不一致。
- **当前处理**：数据库和文件位于同一 `/data` 持久卷，不部署到临时 Serverless 文件系统。
- **补充处理**：项目源文件与素材上传现在按 1 MiB 分块落盘并边写边计数/哈希，不再把 100 MB 文件整体复制到应用内存；仍需在部署层设置磁盘配额和并发策略。
- **后续方向**：S3/R2/OSS 对象存储、预签名上传、文件引用计数与定期清理。

### KI-07：公网 Demo 不等于完整多租户权限系统

- **影响**：当前只有一个管理员身份，不能表达项目归属、团队成员、资源级权限或租户隔离；多实例部署时进程内限流状态也不会共享。
- **当前处理**：应用内管理员登录使用 PBKDF2-SHA256 密码哈希、HttpOnly 会话 Cookie、CSRF 写操作校验、退出和登录失败限流；公网首次部署预置密码哈希并关闭远程首次认领。Caddy 仍默认启用整站 Basic Auth 作为第二层 Demo 入口保护。API 对读写请求分别执行 240/min 与 60/min 的进程内分桶限流，生产部署默认关闭故障注入路由，并限制请求体、容器资源与日志大小。
- **后续方向**：项目归属、团队用户、RBAC、配额、分布式速率限制和审计管理员。

### KI-08：媒体安全检查不等于恶意文件扫描

- **影响**：大型或恶意构造媒体仍可能消耗解码器资源。
- **当前处理**：文件大小、扩展名/MIME 允许列表和安全随机文件名。
- **后续方向**：隔离解码容器、CPU/内存/时间限制、媒体探测和恶意文件扫描。

### KI-09：任务进度采用轮询

- **影响**：任务数量很大时会产生额外请求，事件到达不是毫秒级。
- **当前处理**：Demo 任务数小、更新周期受控，轮询在公网代理下更稳定。
- **后续方向**：服务端事件或 WebSocket，保留轮询作断线回退。

### KI-10：外部 Provider 真实连通性需手动 live 测试验证

- **影响**：默认 CI 用 fake provider 验证协议层（schema、超时、key redact），不验证真实 Gemini、DeepSeek 或 Embedding 端点的连通性和模型名有效性。`llm.py` 对异常会降级回规则；如果只看任务终态而不看运行记录，可能忽略 Provider 实际未成功。
- **当前处理**：`backend/tests/test_llm_live.py`（`@pytest.mark.live`）对真实端点发请求并断言 `degraded=false`；默认被 `-m 'not live'` 跳过。部署或更换 key/模型后应运行 `FRAMEFLOW_RUN_LIVE=1 python -m pytest -m live`。
- **后续方向**：在 staging 环境以定时任务运行 live 测试并告警；保留规则降级作为演示兜底。

### KI-11：无法中断的第三方调用可能需要进程级恢复

- **影响**：Python 线程无法安全强制终止。如果底层 Provider/库忽略网络超时并永久卡住，Worker 只能停止领取新任务，不能在同一进程内回收该线程。
- **当前处理**：每个阶段都有硬超时；超时后任务按重试策略失败或重排，并暂停领取新任务。Worker 会把仍存活的超时执行标记为 `isolated`，`/health/ready` 返回 503、`accepting_jobs=false` 和原因，线程退出后才恢复就绪。
- **后续方向**：把外部调用和重型媒体处理迁移到可终止的独立子进程/专用 Worker，结合进程监督实现真正的强制取消。

### KI-12：预览历史与媒体资源仍需生命周期治理

- **影响**：不同时间线指纹会产生不同的预览记录和 MP4；取消或删除项目期间正在运行的 ffmpeg 可能继续消耗 CPU。当前版本已使用临时输出、执行围栏和失败清理，但不会自动清理项目仍保留的历史成功预览，也没有全局磁盘配额。
- **当前处理**：输出先写入 `.rendering` 临时文件，只有任务围栏通过后才原子发布；备份默认排除可重新下载的模型和缓存，并支持保留天数/数量。
- **后续方向**：增加每项目最新 N 个预览保留策略、全局磁盘配额和可终止的 ffmpeg 进程组。

### KI-13：部署发布验收依赖真实域名与 Docker 环境

- **影响**：本地没有 Docker daemon 或公网 DNS/HTTPS 时，无法完成镜像构建、Caddy 证书、字幕滤镜和公网 smoke 验证。
- **当前处理**：CI 新增 Compose 配置、Shell 语法、Caddy validate、镜像构建和容器 Basic Auth/health smoke；`first-deploy.sh`、`upgrade.sh` 和 `deploy/smoke.sh` 现在会检查应用、Caddy、认证和可选公网 HTTPS。
- **后续方向**：在目标域名执行一次真实部署演练，并将 HTTPS、应用登录、素材删除、预览播放、重启持久化和 acceptance 结果附到发布记录。

## 4. 测试与部署状态

不应把“测试文件存在”等同于“测试已通过”。每次正式提交前需在 `TEST_PLAN.md` 的执行记录中填入实际 commit、环境、日期与结果。

公网地址、测试账号和视频链接只能在真实可访问后写入提交信息，不应在文档中预先伪造。

## 5. 发布前必须更新本文的情况

- 真实 ASR/LLM/Embedding Provider 已配置并有可重复测试证据。
- 身份认证、限流或对象存储的行为再次变化。
- 预览视频、字幕时间轴或媒体生命周期治理再次变化。
- 实际部署不使用持久卷，或使用了多 API/Worker 实例。
- 任何已知问题的影响或缓解方式与实际代码不再一致。
