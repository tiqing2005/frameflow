# FrameFlow AI 4 分 30 秒演示脚本

## 1. 演示目标

在 4 分 30 秒内用一段新文本证明以下事实：

1. 项目创建和耗时处理是真实、持久化的后端流程。
2. 字幕会被分段并生成关键词/主题。
3. 每段至少有 3 个不同候选，且显示命中词和匹配理由。
4. 人工编辑、排序和替换会保存，强制刷新不丢失。
5. 系统可重现任务失败，显示原因并安全重试。
6. 默认运行时如实标识为本地规则/TF-IDF，不伪造外部 AI 或 ASR 结果。

## 2. 录制前 10 分钟检查

- 使用即将提交的 commit，记下 commit SHA。
- 在同一环境启动 API 与 Worker，确认 `/health/live` 和 `/health/ready` 为成功。
- 运行 `scripts/acceptance.ps1` 或 `scripts/acceptance.sh`；如未通过，不开始录制。
- 打开公网演示地址，不用 localhost 冒充公网环境。
- 清除尚未消费的故障注入：演示实验室选择 `none`。
- 确认素材库至少 12 个素材，本地图片无破图。
- 预先复制 `demo/sample-transcript.txt` 和 `demo/failure-retry-transcript.txt`，但不预先创建主演示项目。
- 建议浏览器 1280×720 或 1440×900、90%～100% 缩放，隐藏书签栏和无关通知。
- 在开始前打开一个已完成的 `ai_degrade` 示例运行，仅用于第 7 镜快速说明可追溯性。口播必须说“预先跑过的故障演练”，不假装是当前刚发生。

## 3. 主演示文本

```text
清晨，我骑着自行车穿过城市，准时来到办公室。团队用数据看板梳理项目进度，把复杂任务拆成清晰步骤。午后，我们到公园散步，讨论怎样用绿色科技让工作更高效，也让生活更健康。
```

该文本故意覆盖城市/交通、团队/数据/办公、自然/健康/绿色科技三组语义，便于在短时间内直观检查排序是否随内容变化。

## 4. 4:30 逐镜头时间表

### 镜头 1｜00:00–00:18｜定位与业务闭环

**画面**：从项目台开始，画面同时可见 FrameFlow AI 名称、四个主导航和“新建项目”。鼠标掠过项目/素材/运行数指标，不进入详情。

**口播**：

> FrameFlow AI 是一个字幕到素材的轻量工作台。这次我聚焦真实闭环：输入、异步处理、可解释匹配、人工修正和持久化，不做一个表面完整但无法稳定验收的剪辑器。

**证据点**：产品范围清晰，首屏有真实服务端数据。

### 镜头 2｜00:18–00:48｜用新文本创建项目

**画面**：点击“新建项目”。快速切换一次“上传音视频” Tab，展示真实文件选择入口与 ASR 配置提示，然后回到“粘贴文本”。填写名称“演示－绿色高效的一天”，粘贴主文本，点击创建。

**口播**：

> 系统同时接受音视频和文本。默认演示未配置外部 ASR，所以音视频会真实上传，但不会伪造转写结果；文本是稳定主入口。这里提交一段带城市、数据和绿色科技的新文本。

**证据点**：上传 UI 真实存在，但口播没有夸大 ASR 能力；新文本不是程序内写死字符串。

### 镜头 3｜00:48–01:10｜真实异步进度

**画面**：创建返回后自动进处理页。停留，让录屏看到至少 `validating/segmenting/keywording/matching/persisting` 中的多个真实事件。

**口播**：

> API 只在事务里创建项目和持久化任务，然后返回 202。独立 Worker 从 SQLite 原子领取任务。这些阶段、进度和事件都来自服务端，刷新或进程重启后不会因前端内存丢失。

**证据点**：HTTP 202 语义、独立 Worker、服务端事件。

### 镜头 4｜01:10–02:00｜分段与可解释候选

**画面**：进入工作台。从左栏指向原始字幕与 3 个语义片段。切换“城市骑行”、“数据看板”和“绿色科技”片段，每次在右栏滑过前 3 个候选。展开其中一张的分项或解释。

**口播**：

> 文本被通用标点和长度规则切分，每段提取主题和关键词。右侧每段都有至少三个不同候选。排序不是只给一个神秘分数：55% 是字符 n-gram TF-IDF 向量相似度，30% 是关键词重合，15% 是标签和主题重合。这里可以看到命中词和中文理由；如果是补位候选，也会明确标记。

**证据点**：原文、通用分段、关键词、三候选、分项与理由。

### 镜头 5｜02:00–02:35｜人工编辑与排序

**画面**：选中第二片段，在文本末补入“并持续检查风险。”，将主题改为“数据协作”或增加关键词“风险”，保存。展示“保存中→已保存”。将该片段与相邻片段调整顺序，等待排序保存反馈。

**口播**：

> 自动分析是初稿，人才是最终决策者。文本修改带版本号，可以防止另一个页面静默覆盖结果。顺序是整体事务保存，不会留下重复位置。

**证据点**：可编辑、保存反馈、乐观锁思路、排序事务。

### 镜头 6｜02:35–03:10｜搜索替换与刷新证明

**画面**：选中“绿色科技”片段，在右栏全库搜索“自然”或“健康”，选一张不是当前自动选择的素材，点击替换。等待“已保存”与“手动选择”标记。按 Ctrl+R 或浏览器刷新，回到同一片段，指向刚才文本、顺序和素材。

**口播**：

> 除了三个候选，我还可以搜索整个素材库快速替换。用户选择会保存为 manual，之后的重新匹配也不会静默覆盖。现在强制刷新，可以看到文本、排序和手工选择都从 SQLite 重新加载。

**证据点**：搜索、快速替换、manual 优先、刷新持久化。

### 镜头 7｜03:10–03:35｜素材与运行追溯

**画面**：新窗口或导航快速进入素材库，展示至少 12 个本地素材和搜索。随后进入“AI 运行记录”，指向当前项目的 `provider=rules`、策略版本和一条预先跑过的 degraded 记录。

**口播**：

> 素材库预置了十二个可离线访问的素材，也支持真实上传和元数据搜索。运行记录如实显示当前 Provider 是 rules，TF-IDF 是本地稀疏向量，不是神经 Embedding。这条降级记录是我录制前用可重复故障入口跑出的，不代表真实外部 Provider 刚才宕机。

**证据点**：12 素材、本地资源、Provider 真实标识、降级追溯。

### 镜头 8｜03:35–04:15｜可重复失败与重试

**画面**：进入“演示实验室”，指向“这是故障演练”警告。点击“下一次任务失败”，立即新建文本项目“故障演练”，粘贴 `failure-retry-transcript.txt`（若时间紧，只粘贴前两句）。看到 failed 状态、错误原因和重试按钮后点击重试，等待至少进入 running；如处理速度足够，停到 succeeded。

**口播**：

> 这是明确标注的一次性故障演练，用来稳定验收失败路径。任务失败后保留已上传的输入、错误码、失败阶段和事件历史。点击重试会增加尝试次数并重新入队，不是又创建一个项目。故障只消费一次，所以重试可正常完成。

**证据点**：故障明确标注、持久化 failed、原因、事件历史、重试语义。

### 镜头 9｜04:15–04:30｜取舍和结束

**画面**：回到主项目工作台，定格在字幕、预览和三候选同时可见的画面。

**口播**：

> 这个 Demo 的边界也是明确的：它是单实例 SQLite 与单 Worker，默认没有真实 ASR，也不生成预览视频。但核心闭环、异常恢复、可解释匹配和人工结果持久化都是可以独立复现的。

**证据点**：主动披露边界，不夸大产品能力。

## 5. 时间超支处理

- 若主任务处理超过 22 秒，口播继续解释当前事件，不剪掉等待并伪装瞬时完成。可缩短第 7 镜。
- 若故障重试未在 04:15 前完成，只要已显示 queued/running 且尝试次数增加，即可进入结束语；录制后需单独复验它最终 succeeded。
- 若公网环境当场出现非注入失败，保留错误画面并说明 request_id，再根据可重试性操作。不用前端假数据跳过。

## 6. 演示中禁止的说法

- “这里调用了大模型”，如果运行记录实际为 `rules`。
- “这是语义 Embedding”，如果实际为字符 n-gram TF-IDF。
- “支持音视频自动转写”，如果演示环境没有真实 ASR Provider。
- “已支持高并发”，当前是单 Worker 可靠排队而非高吞吐。
- “这是真实 AI 宕机”，对一次性故障注入应说“可重复故障演练”。

## 7. 录制后校验

- 时长在 4:15～4:45 之间，目标 4:30。
- 项目创建、处理事件、三候选、人工替换、强制刷新、故障和重试画面都在。
- 全程无 API Key、`.env`、个人账号、私有仓库 URL 或本地绝对路径。
- 画面可看清错误原因和匹配理由，不只靠口播。
- 视频描述附上公网地址、源码地址、测试账号（如有）和录制对应 commit。
